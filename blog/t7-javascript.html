How does Ruby and JavaScript differ?

Sunday, February 8th, 2015

- Ruby classes vs. JavaScript constructor functions - Ruby documentation vs. JavaScript documentation - Looping in Ruby vs. Looping in JavaScript

-Welcome back!!!! On this episode, I will be discussing the some of the similarities and differences between the Ruby and JavaScript languages. Here are some of the topics I'll be covering:

  • - Ruby Hashes vs. JavaScript Objects
  • - Ruby classes vs. JavaScript constructor functions

- Ruby Hashes vs. JavaScript Objects

In the beginning of January, we took a look at the differences between "hashes" and "arrays". I gave the following example:
hash = {
key1 => value1,
key2 => value2,
key3 => value3
}
puts hash["key1"]
puts hash["key2"]
puts hash["key3"]
A "hash" is, essentially, a list of items. Each item in the list has a corresponding value to it. A way to look at it in non-computer related terms is by thinking of it like a shopping list of items which has the name of the item and the quantity needed. for example: eggs => a dozen, socks => 3 pair, coffee => 1lbs, milk => 2 gallons etc. As you know, each item is referred to as an "item" on the list and the quantity you know as the "quantity". In the ruby language, if that shopping list was a hash, each "item" would be known as a "key" and each "quantity" of each item would be known as a "value" of each "key". The handy thing about this type of list (a hash) is that when used in the ruby language, both the keys and values can be used in many different ways.

In JavaScript, "Objects" have a somewhat similar nature. In that language, an object, much like the "keys" in a hash, has a value. There are differences tho. According to the W3, declaring strings, numbers, and boolean objects has an unwanted reaction. Apparently, the processing speed gets diminished when attempting to declare those examples. Additionally, whereas in Ruby hashes are essentially a collection of instance variables that have limited access, in JavaScript there is no need for such restrictions. Here's an example of a JavaScript object:

var person = {
firstName:"John",
lastName:"Doe",
age:50,
eyeColor:"blue"
};

- Ruby classes vs. JavaScript constructor functions

A Ruby Class, is a first-class object. The fact is that Ruby and JavaScript are both object oriented languages. This means that they create objects and use them as tools to run programs that are aimed at one or more functions. You can think of an object, in reference to a car, as the different parts of a car that allow you to drive it and control its movements. So in that sense, lets say that we have a ruby class called Car. This would be declared at the top of the program that you create. Like so:

  • class Car
  • def initialize(wheels, gas)
  • @wheels = wheels
  • @gas = gas
  • end

    • In this example the class 'class Car' has some attributes that you probably recognize. Though, this isn't exactly what the full program might be, it does cover the main principles that we're looking to explain. These attributes can be used later in the continued portion of the program in order to implement and access functions. So let's compare this ruby class to what it might be in JavaScript.

    • function car(wheels, gas) {
    • this.wheels = wheels;
    • this.gas = gas;
    • @gas = gas
    • }

      • The differences are that in Ruby and JavaScript, the class "class Car" and its attributes are accessed in a slightly different manner. Obviously the syntax is going to be different. However, ultimately they can both achieve a similar level of practicality. For each of them, we can distribute values, and create an almost endless list of instances of them. An instance, is another way of saying " a running example of it's capabilities".
